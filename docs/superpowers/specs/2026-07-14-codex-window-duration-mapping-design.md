# Codexレート制限ウィンドウの時間長分類設計

日付: 2026-07-14

対象: Token CheckerのCodex使用量取得

## 背景

Codexの使用量APIは、以前は5時間枠を`primary_window`、週次枠を`secondary_window`として返していた。

現在のレスポンスでは5時間枠が一時的に返らず、週次枠が`primary_window`へ移動している。
この週次枠は`limit_window_seconds`が`604800`であり、ウィンドウ自体の時間長は判別できる。

現実装はフィールドの位置だけで5時間枠と週次枠を割り当てるため、週次使用率を5時間欄へ表示し、週次欄をデータなしとして表示する。

## 表示仕様

5時間枠がAPIレスポンスに含まれない場合も、Codexカードの5時間欄は残す。
この欄には既存の「データなし」を表示する。

週次枠が`primary_window`または`secondary_window`のどちらに含まれていても、週次欄へ使用率、残り時間、リセット日時、経過セグメントを表示する。

ポップオーバーの2列構成とメニューバーの2段ミニバーは変更しない。
メニューバーでは、5時間枠に対応するミニバーを未取得状態のトラックとして表示し、週次枠に対応するミニバーへ週次使用率を表示する。

## ウィンドウの分類

OAuth使用量クライアントは、各ウィンドウの`limit_window_seconds`を読み取る。

分類規則は次のとおりとする。

- `18000`秒：5時間枠
- `604800`秒：週次枠
- それ以外：未対応の時間長として無視

`primary_window`と`secondary_window`を順に走査し、フィールド名ではなく時間長で`ServiceUsage`へ割り当てる。

同じ時間長のウィンドウが複数含まれる場合は、レスポンス内で先に現れるウィンドウを採用する。
これは重複レスポンスを誤って上書きしないための決定であり、通常レスポンスの表示には影響しない。

`limit_window_seconds`が欠けているウィンドウは分類できないため無視する。
位置からの推測は、今回と同じ誤分類を再発させるため行わない。

## 既存取得経路との関係

OAuth取得経路だけを変更する。

互換経路であるCodex app-serverクライアントは、すでに`windowDurationMins`を使って5時間枠と週次枠を分類しているため変更しない。

認証エラー、通信エラー、デコードエラーに対する既存のフォールバック方針も変更しない。
未知の時間長や特定ウィンドウの欠損はレスポンス全体の取得失敗とはせず、対応する`RateLimit`を`nil`にする。

`additional_rate_limits`は今回の表示対象に含めない。

## テスト

OAuth使用量クライアントの単体テストへ次のケースを追加する。

1. 従来形式として、5時間枠が`primary_window`、週次枠が`secondary_window`にある場合、両方を正しく分類する。
2. 現在の形式として、週次枠が`primary_window`にあり、`secondary_window`が`null`の場合、`fiveHour`を`nil`、`weekly`を週次使用率として返す。
3. 未対応の`limit_window_seconds`だけが含まれる場合、そのウィンドウを5時間枠や週次枠へ誤分類しない。

実装は失敗するテストを先に追加し、期待した理由で失敗することを確認してから行う。

実装後は`swift test`、`swift build`、`git diff --check`を実行する。

## 受け入れ条件

1. 5時間枠がレスポンスから欠けているとき、Codexの5時間欄に「データなし」と表示される。
2. `primary_window`に入った週次枠がCodexの週次欄へ表示される。
3. 従来の5時間枠と週次枠を含むレスポンスも正しく表示される。
4. 未対応の時間長を5時間枠または週次枠として表示しない。
5. Claudeの表示、ポップオーバーの配置、メニューバーの構成、OAuthのフォールバック方針を変更しない。
6. `swift test`、`swift build`、`git diff --check`が成功する。
